# ADR-0001: аутентификация и владение данными (v1.1)

## Контекст

В v1.0 API было полностью открытым: любой клиент видел и мог удалить
любые задачи и лиды. Согласно roadmap (раздел Core), это первый пробел,
который нужно закрыть — все последующие модули (Company/Project
management, персональный dashboard, workflow) неявно предполагают, что
данные принадлежат конкретному пользователю.

## Решение

1. **Аутентификация — JWT (HS256), выпускается и проверяется собственной
   реализацией на стандартной библиотеке** (`internal/auth`), без
   сторонних JWT- или crypto-библиотек. Причины:
   - требования проекта — один алгоритм подписи, фиксированный набор
     claims — полностью покрываются `crypto/hmac` + `encoding/json`;
   - принцип проекта "не вводить технологии без явной необходимости";
   - дополнительное практическое ограничение текущего окружения сборки:
     Go module proxy недоступен из песочницы, где готовился этот
     инкремент, что сделало добавление новых модулей ненадёжным решением
     прямо сейчас.

   Если в будущем понадобятся возможности за пределами простого HS256
   (refresh-токены с ревокацией, асимметричные ключи, JWKS для внешних
   потребителей), стоит пересмотреть это решение в пользу `golang-jwt/jwt`.

2. **Хэширование паролей — PBKDF2-HMAC-SHA256** (`internal/auth/password.go`),
   210 000 итераций, соль 16 байт, вместо bcrypt/argon2. Та же причина:
   обе стандартные альтернативы живут в `golang.org/x/crypto`, а не в
   стандартной библиотеке. Формат хранения хэша самоописывающийся
   (`pbkdf2-sha256$iterations$salt$hash`), что оставляет путь миграции на
   другую схему хэширования без инвалидации всех существующих паролей.

3. **Модель владения:** `tasks.user_id` — nullable FK на `users` (nullable
   ради безопасной миграции существующих данных; на уровне сервиса
   всегда проставляется для новых задач). `leads` не имеет собственной
   колонки `user_id` — владение наследуется через `task_id`, все выборки
   лидов идут через `JOIN` с `tasks`.

4. **Область защиты:** `/api/v1/auth/register` и `/api/v1/auth/login`
   остаются публичными (иначе получить первый токен неоткуда).
   `/api/v1/auth/me`, `/api/v1/tasks/*` и `/api/v1/leads/*` требуют
   валидный Bearer-токен.

## Не входит в этот инкремент

- Refresh-токены и отзыв токенов (logout только на клиенте, сам JWT
  до истечения TTL остаётся валидным).
- RBAC/права на основе `role` (поле есть в модели, но пока ни на что не
  влияет — задел на будущее, когда появится Company/Project management
  с разграничением доступа внутри команды).
- OAuth/SSO.
- Email-подтверждение при регистрации.

## Последствия

- Новый пользователь регистрируется через `POST /api/v1/auth/register` и
  сразу получает токен — есть только один способ создать первого
  пользователя, отдельного "seed"-механизма нет.
- Локальный запуск "из коробки" по-прежнему работает без переменных
  окружения: `auth.jwt_secret` имеет dev-значение по умолчанию, но при
  старте пишется явное предупреждение в лог, если оно не переопределено.
